2. HIGH-LEVEL PRODUCT DESCRIPTION
========================================

Build a web app for **Rent-A-Buddy™**, where:

- **Clients** can:
  - Sign up / log in
  - Complete a simple client profile
  - Browse Buddies
  - View Buddy profiles
  - Request bookings (sessions)
  - View upcoming/past bookings
  - Leave reviews after sessions
  - Accept a Client Safety Agreement during onboarding

- **Buddies** can:
  - Sign up as Buddies
  - Complete a Buddy profile (bio, city, hourly rate, skills, interests)
  - See a Buddy Certification checklist
  - Set availability (simple weekly slots)
  - See incoming booking requests
  - Accept/decline bookings
  - View their schedule of confirmed sessions
  - Accept Code of Conduct & Safety Protocol

- **Admins** can:
  - Log in as admin
  - See dashboards for users, bookings, verifications, incidents
  - Approve/reject verifications
  - Suspend/activate users
  - Update status of incident reports

This is NOT a dating app and NOT an escort service.  
The UI and copy must clearly emphasize **safe, non-romantic companionship**.

4. DATABASE SCHEMA (PRISMA MODELS)
========================================

Use Prisma with SQLite and define at least these models:

- User
  - id
  - name
  - email (unique)
  - passwordHash
  - role: "CLIENT" | "BUDDY" | "ADMIN"
  - status: "ACTIVE" | "SUSPENDED"
  - createdAt
  - updatedAt

- ClientProfile
  - id
  - userId (FK → User)
  - city
  - shortBio
  - safetyAgreementAcceptedAt (DateTime)

- BuddyProfile
  - id
  - userId (FK → User)
  - headline
  - bio
  - city
  - hourlyRate (Float or Int)
  - experienceYears (Int)
  - languages (string or JSON field)
  - tags (string or JSON)
  - ratingAverage (Float)
  - ratingCount (Int)
  - isCertified (Boolean)
  - codeOfConductAcceptedAt (DateTime)
  - safetyProtocolAcceptedAt (DateTime)

- BuddyAvailability
  - id
  - buddyProfileId (FK → BuddyProfile)
  - dayOfWeek (Int 0–6 or string)
  - startTime (string, e.g. "09:00")
  - endTime (string, e.g. "12:00")
  - isActive (Boolean)

- Verification
  - id
  - userId (FK → User)
  - type: "ID" | "BACKGROUND_CHECK" | "CERTIFICATION" | "OTHER"
  - status: "PENDING" | "APPROVED" | "REJECTED"
  - submittedAt
  - reviewedAt
  - reviewedByAdminId (FK → User)
  - notes

- Booking
  - id
  - clientId (FK → User)
  - buddyId (FK → User)
  - startTime (DateTime)
  - endTime (DateTime)
  - status: "PENDING" | "CONFIRMED" | "COMPLETED" | "CANCELED" | "REJECTED"
  - locationType: "PUBLIC" | "VIRTUAL"
  - locationDescription (string)
  - clientNotes (string)
  - buddyNotes (string)
  - createdAt
  - updatedAt

- Review
  - id
  - bookingId (FK → Booking)
  - clientId (FK → User)
  - buddyId (FK → User)
  - rating (Int 1–5)
  - comment (string)
  - createdAt

- Message (simple session chat)
  - id
  - bookingId (FK → Booking)
  - senderId (FK → User)
  - content (string)
  - createdAt

- IncidentReport
  - id
  - bookingId (FK → Booking)
  - reporterId (FK → User)
  - reportedUserId (FK → User)
  - type: "SAFETY" | "BOUNDARY" | "BEHAVIOR" | "OTHER"
  - description (string)
  - status: "OPEN" | "UNDER_REVIEW" | "RESOLVED"
  - createdAt
  - updatedAt
  - handledByAdminId (FK → User)

You can add small helper tables if absolutely necessary, but keep this MVP lean.

5. FEATURES TO IMPLEMENT (MVP)
========================================

A. Authentication & Authorization
---------------------------------
- User registration (client or buddy)
- Login / Logout
- Password hashing
- JWT or sessions for auth
- Middleware to protect:
  - Client dashboard routes
  - Buddy dashboard routes
  - Admin routes

B. Client Features
------------------
- Client onboarding flow:
  - After signup, complete ClientProfile with city & short bio
  - Accept Client Safety Agreement (checkbox + store timestamp)
- Browse Buddies:
  - Simple filter by city and optionally hourly rate
- Buddy detail page:
  - Show profile, bio, hourly rate, rating, availability
- Create booking request:
  - Select Buddy, date/time range within their availability
  - Choose locationType: PUBLIC or VIRTUAL
  - Add short notes
- View “My Bookings”:
  - Tabs: Upcoming, Past
  - Click into booking details
- Post-session review:
  - After COMPLETED booking, client can submit rating + comment

C. Buddy Features
-----------------
- Buddy onboarding flow:
  - After signup as Buddy, complete BuddyProfile (headline, bio, city, hourlyRate, etc.)
  - Accept Buddy Code of Conduct + Safety Protocol (checkbox + store timestamps)
- Availability management:
  - Weekly schedule table (day-of-week + time ranges)
- Booking requests:
  - View PENDING bookings
  - Accept or Reject
- Sessions list:
  - View CONFIRMED/COMPLETED bookings in a simple chronological list

D. Admin Features
-----------------
- Admin dashboard:
  - Summary counts for Users, Buddies, Bookings, IncidentReports
- Users management:
  - Table of users with filters by role and status
  - Ability to change status (ACTIVE/SUSPENDED)
- Verifications:
  - List of pending verifications
  - Approve/Reject
- Incident reports:
  - View list of incident reports
  - Change status and add resolution notes

E. Safety & Policy UX
---------------------
- During signup/onboarding:
  - For Clients: show **Client Safety Agreement**, **Zero Sexual Content Policy**, **Anti-Escort Clause**
  - For Buddies: show **Buddy Code of Conduct**, **Buddy Safety Protocol**, **Zero Sexual Content Policy**, **Anti-Escort Clause**
- Store acceptance timestamps in the DB.
- Display a “Non-romantic, non-escort service” disclaimer in the footer and relevant screens.

F. Messaging (MVP)
------------------
- For each booking, allow messages between client and buddy.
- Display messages on a Booking Detail page.
- Simple send box + message list ordered by createdAt.
